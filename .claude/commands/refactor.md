# /refactor

外部から見た動作を保ったまま、コードの内部構造を改善します。

## 使い方

```
/refactor src/utils/parser.ts が長すぎるので分割したい
/refactor 重複している API 呼び出し処理を共通化したい
```

## 手順

1. **安全網の確認**: 対象範囲のテスト有無を確認する。テストがなければ、先に現在の動作を固定するテスト（characterization test）を書くことを提案する
2. **現状分析**: 対象コードの問題点（重複・長すぎる関数・混在する責務）を列挙する
3. **計画提示**: 変更ステップを提示する。1ステップ = 1つの改善 = テストが通る状態を保つ
4. **段階実行**: ステップごとにテストを実行しながら進める
5. **完了報告**: 動作が変わっていないことをテスト結果で示す

## 出力形式

```
## 対象と問題点
- path/to/file.ts — 問題の内容

## リファクタリング計画
1. ...（各ステップ後にテスト実行）

## 実施結果
- 変更ファイル: ...
- テスト: 前 X件パス → 後 X件パス（増減なし）
```

## 完了条件

- [ ] 全テストが変更前と同じ結果でパスしている
- [ ] 機能追加・バグ修正が混ざっていない
- [ ] 途中で見つけたバグは修正せず報告している

## 注意点

- **リファクタリングとバグ修正を混ぜない**。途中でバグを見つけたら報告し、別タスク（/debug）に分ける
- 悪い例: `ついでにエラーハンドリングも追加しておきました` ／ 良い例: `構造の変更のみ。気づいた問題は報告欄に記載`
- テストがない領域に触るときは、まずテストを書いてから触る
