# /api-design

API の契約（エンドポイント・スキーマ・エラー仕様）を設計します。実装より先に契約を固めます。

## 使い方

```
/api-design ユーザーのお気に入り登録・解除・一覧
/api-design 既存の /api/logs を見直したい
```

## 手順

1. **要件整理**: 誰が・何のために・どんなデータを扱うかを確認する
2. **既存調査**: 既存 API の命名規則・認証方式・エラー形式を調べ、一貫性を保つ
3. **設計**: エンドポイント・リクエスト/レスポンススキーマ・エラーケースを定義する
4. **自己チェック**: 完了条件のチェックリストで自己レビューする
5. **提示**: ユーザーの承認を得てから実装に進む（実装は別コマンド）

## 出力形式

```
## エンドポイント一覧
| メソッド | パス | 目的 | 認証 |

## スキーマ定義
（TypeScript 型 or OpenAPI。プロジェクトの慣習に合わせる）

## エラー仕様
| ステータス | 条件 | レスポンス例 |

## 設計判断とトレードオフ
...
```

## 完了条件

- [ ] 正常系と異常系（バリデーション・認証・404 等）が両方定義されている
- [ ] 既存 API との命名・構造の一貫性を確認済み
- [ ] 一覧系はページネーション、更新系は冪等性に言及している

## 注意点

- 悪い例: `POST /getUserData`（パスに動詞） ／ 良い例: `GET /users/{id}`
- レスポンスに内部実装の都合（DB カラム名そのまま等）を漏らさない
- 既存 API の破壊的変更になる場合は必ず明示し、移行方針を確認する
