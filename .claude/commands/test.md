# /test

対象コードのテストを設計し、追加します。

## 使い方

```
/test src/lib/growth.ts の判定ロジック
/test 先ほど実装したエクスポート機能
```

## 手順

1. **対象把握**: 対象コードの責務と入出力を把握する
2. **ケース設計**: 正常系・境界値・異常系を列挙し、優先度をつける
3. **既存確認**: 既存テストの書き方（フレームワーク・命名・モック方針）に合わせる
4. **実装**: 優先度の高いケースからテストを書く
5. **実行**: 全テストを実行し、追加分がパスすることを確認する

## 出力形式

```
## テストケース設計
| ケース | 種別 | 優先度 |
|--------|------|--------|
| 正しい入力で期待値を返す | 正常系 | 高 |
| 空配列で空を返す | 境界値 | 高 |
| 不正な型でエラーを投げる | 異常系 | 中 |

## 追加したテスト
- path/to/file.test.ts — N件

## 実行結果
X件パス / 0件失敗
```

## 完了条件

- [ ] 正常系だけでなく境界値・異常系が含まれている
- [ ] 全テストがパスしている
- [ ] テストが内部実装ではなく振る舞い（入力→出力・副作用）を検証している

## 注意点

- 悪い例: 内部メソッドの呼び出し回数だけを検証する ／ 良い例: 入力に対する出力と観測可能な副作用を検証する
- テストのためだけに本体コードを変えない（必要なら理由を報告して承認を得る）
- 通らないテストを skip で隠さない。通らない理由を報告する
